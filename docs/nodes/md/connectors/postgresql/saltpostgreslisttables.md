# PostgreSQL List Tables

<div style="display: flex; gap: 20px; align-items: flex-start; margin-bottom: 20px;">
<div style="flex: 1; min-width: 0;">

This node connects to a PostgreSQL database using a stored PostgreSQL credential and lists all tables in a specified schema. It calls a `/list-tables` operation on the backing PostgreSQL service and then formats the response using shared database utilities. The node returns both a human-readable summary and a JSON-encoded payload that can drive downstream query-building, exploration, or automation steps.

## Usage

Use this node when you need to discover which tables exist in a particular PostgreSQL schema before writing queries, configuring visual query builders, or orchestrating data workflows. Place it early in your pipeline after you have created a PostgreSQL credential in Salt (using the `postgres` credential template) and, optionally, validated connectivity with a simple query node such as `PostgreSQL Query`. You provide three main inputs: a `credentials_path` pointing to the credential configuration, a `timeout` in seconds for the backend call, and the target `schema` name (for example, `public`, `analytics`, or `sales`). The node will then list all tables visible to that credential within the schema. A typical pattern is: credentials setup → `PostgreSQL List Tables` → user or LLM chooses a table → `PostgreSQL Table Info` or `PostgreSQL Query` for deeper inspection; or credentials setup → `PostgreSQL List Tables` → send `json_output` into `PostgreSQL Visual Query Builder` or `PostgreSQL Query Builder Helper` to populate table dropdowns and inform join suggestions. Rely on the JSON output for machine processing, keep the timeout aligned with your database size and network latency, and run the node once per schema if you need a multi-schema catalog.

## Important Notes
- **Performance**: Listing tables in large schemas can be slow; configure `timeout` with enough headroom and avoid invoking this node repeatedly in tight loops across many schemas.
- **Limitations**: Each execution only lists tables from a single schema. To build a multi-schema catalog, call the node once per schema and merge results in a downstream step.
- **Behavior**: The exact JSON shape (field names such as `table_name`, presence of `row_count`, and any extra metadata) is determined by the shared DatabaseBaseNode and the PostgreSQL service configuration. Inspect sample output in your environment before binding downstream parsing to specific fields.
- **Permissions**: The list only includes tables that the authenticated PostgreSQL user is allowed to see. Tables in schemas where the user lacks metadata or schema privileges will not appear, even if they exist in the database.
- **Service dependency**: This node relies on an external PostgreSQL service endpoint (`/list-tables`); if that service is misconfigured or unavailable, the node will fail regardless of your input values.

## Troubleshooting
- **Empty or unexpectedly small table list**: If the node runs without explicit errors but returns no or too few tables, confirm that the `schema` value is correct (including case) and verify that the PostgreSQL user in your credential has USAGE and metadata privileges on that schema.
- **Timeout or connection errors**: Errors mentioning timeouts, unreachable hosts, or connection failures typically indicate incorrect host/port or credentials, network restrictions, or too low a `timeout`. Validate the credential at `credentials_path`, test with a simple PostgreSQL query node, and increase `timeout` if necessary.
- **Permission or authorization failures**: If you see permission-denied or authorization errors, adjust the PostgreSQL user’s privileges so it can list tables in the target schema, or switch to a credential with sufficient access.
- **Downstream JSON parsing failures**: When consumer nodes fail while parsing the output, ensure they are using the `json_output` field and parsing it exactly once as JSON. Parsing `text_output` as JSON or double-decoding the JSON string will result in errors.
